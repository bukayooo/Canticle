# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Canticle is an iOS devotional app built on the 1662 Book of Common Prayer. On launch it checks the
clock and calendar and shows today's Morning or Evening Prayer — fixed liturgy plus the day's
appointed Psalms and Lessons — in a single scrollview. See `README.md` for the full story of every
content source and the tradeoffs behind it; this file is oriented toward getting productive in the
code quickly.

## Commands

The Xcode project (`Canticle/Canticle.xcodeproj`) is **generated by
[XcodeGen](https://github.com/yonaskolb/XcodeGen) from `Canticle/project.yml`** — do not hand-edit
`project.pbxproj` for anything that `project.yml` can express (targets, sources, build settings,
whole-directory resources). After changing `project.yml`:

```bash
cd Canticle && xcodegen generate
```

**This also applies to `Resources/` contents.** `project.yml`'s `resources:` entries
(`Canticle/Resources`, `Canticle/Preview Content`) are enumerated into individual file references
at generation time, not watched live — adding, removing, or renaming any file under
`Canticle/Canticle/Resources` leaves the `.xcodeproj` stale until `xcodegen generate` runs again. A
stale reference to a deleted file fails the build ("No such file or directory"); a new file just
silently isn't compiled/copied until regenerated.

Build from the CLI (matches what Xcode itself runs):

```bash
xcodebuild -scheme Canticle -configuration Debug \
  -destination 'platform=iOS Simulator,id=<SIMULATOR_UDID>' \
  -workspace Canticle/Canticle.xcodeproj/project.xcworkspace build
```

There is no test target/scheme in this project (no XCTest files, nothing under `schemes:` for
testing in `project.yml`) — don't invent a `xcodebuild test` invocation.

**`Resources/BibleOriginal` and the eight `Icon1.icon`–`Icon8.icon` bundles are genuine Xcode
*folder references*** (blue folders, `type: folder` in `project.yml`), each paired with an
`excludes` entry on its containing directory's `sources:`/`resources:` entry (the generic top-level
`- path: Canticle` entry *and* the sibling `Canticle/Resources` entry both need the exclude — a
directory declared as its own `sources:`/`resources:` entry is not automatically deduped against an
overlapping parent entry in the same list). This is required because `Resources/Bible` and
`Resources/BibleOriginal` share basenames (both have a `genesis.json`), and each `Icon*.icon`
bundle's internal `Assets/*.png` files can share names across bundles — without the folder
reference + excludes, XcodeGen's default enumeration flattens everything to the bundle root and
same-named files collide ("Multiple commands produce .../genesis.json"). If you add another
directory that needs `type: folder` treatment, exclude it from every overlapping parent entry the
same way, then verify with a duplicate-reference check (search the generated `.pbxproj` for `in
Resources */,` / `in Sources */,` lines and confirm no filename appears twice) before shipping.

**⚠️ Known landmine**: XcodeGen 2.43.0 predates Xcode's Icon Composer feature and has no extension
mapping for `.icon` bundles, so `type: folder` on `Icon1.icon`–`Icon8.icon` produces the generic
`lastKnownFileType = folder` instead of the `folder.iconcomposer.icon` UTI that `actool` requires to
recognize them as app-icon sources. Without the correct UTI the build fails at
`CompileAssetCatalogVariant` with `"None of the input catalogs contained a matching ... app icon set
... named Icon1."` **After every `xcodegen generate`, also run**:

```bash
ruby Scripts/fix_icon_composer_file_types.rb
```

This patches the UTI back via the `xcodeproj` gem; it's idempotent and safe to re-run. `project.yml`
also can't express `DEVELOPMENT_TEAM` or the real `PRODUCT_BUNDLE_IDENTIFIER` drifting if someone
changes code signing from Xcode's GUI (which writes straight to `project.pbxproj`, bypassing
`project.yml`) — if you change signing in Xcode, mirror the change into `project.yml`'s
`settings.base`, or the next `xcodegen generate` will silently revert it and the next build will
install as a second, separately-identified app instead of updating the existing one.

Useful one-off scripts under `Scripts/` (not part of the app build, not run automatically):
- `convert_original_language_bible.py` — regenerates all of `Resources/BibleOriginal` (Hebrew OT,
  Greek NT, Greek/Latin Apocrypha) from source corpora. Requires `pip install betacode pygtrie`.
  Reads from `~/Downloads/Old Testament`/`~/Downloads/New Testament` and fetches/caches the
  Septuagint (CCAT/CATSS) and Latin 2 Esdras (vulgate.org) over the network.
- `add_bible_original_to_xcodeproj.rb` — the one-off script that originally added the
  `BibleOriginal` folder reference by hand, before `project.yml` was updated to express it
  natively; kept for history, not needed for a normal `xcodegen generate` anymore.
- `fix_icon_composer_file_types.rb` — required after every `xcodegen generate`; see the landmine
  above.

## Architecture

**Data flow**: bundled JSON in `Resources/` → a `@MainActor` `ObservableObject` store per content
type (loads on demand, not eagerly) → `DevotionalViewModel` merges everything for "today" into a
flat `[DisplayBlock]` → `ContentView` renders it in one `ForEach` inside a single `ScrollView`. There
is no navigation between days/screens for the main experience — the app always shows *today's*
office; Settings, the Litany, and A Commination are the only other screens.

**Stores** (`Services/`), each independently loading its own bundled JSON, keyed by different
lookups because the underlying liturgical data has genuinely different shapes:
- `TimeOfDayService` — no state; a pure function of the clock deciding Morning vs. Evening and
  computing `dayOfYear`/`dayOfMonth`.
- `CalendarStore` — `Resources/Calendar/daily_readings.json`, keyed by `dayOfYear` (1–365; leap years
  are a known, deliberate off-by-one after Feb 29 — see README).
- `PsalterStore` — `Resources/Psalter/{psalms,psalms_hebrew,psalter_cycle}.json`, keyed by
  `dayOfMonth` + `Office` (the 1662 Psalter's 30-day cycle doesn't depend on the calendar date at
  all, unlike everything else).
- `BibleStore` — `Resources/{Bible,BibleOriginal}/*.json`, one file per book, loaded lazily and
  cached; resolves free-text Lesson references (`ScriptureRef`) to verse text. Also holds the
  `useOriginalLanguages` toggle (see below).
- `ProperSundayStore`, `CollectsStore` — Sunday-specific Lessons/Collects, resolved via
  `ChurchYear.namedSunday(for:)`/`governingSunday(for:)`, which need `MovableFeasts`' Easter computus
  for the Easter-anchored portion of the year.

**`DevotionalViewModel.rebuild()`** is the seam where all of the above gets pulled together for a
given `DayContext` (date + office); it's called from `init`, `refresh()` (app foreground), the
DEBUG office switcher, and a Combine subscription to `BibleStore.$useOriginalLanguages` (needed
because `@Published`'s projected publisher fires in `willSet`, before the new value is actually
stored — the subscription defers the rebuild via `Task { @MainActor in ... }` so it re-reads the
*new* value rather than the stale one still in place mid-`willSet`). Any store whose output can
change without a new `DayContext` needs the same treatment.

**Missing content is explicit, never fabricated**: if a Lesson reference doesn't parse, a passage
isn't bundled, a Sunday doesn't resolve, etc., `DevotionalViewModel` emits
`.missingDailyContent(kind:)` and `MissingDailyContentView` renders a placeholder — this is a
deliberate app-wide convention, not an oversight to "fix" by guessing at content.

### Original-language Bible text (Hebrew/Greek/Latin)

`BibleStore.useOriginalLanguages` (persisted via `UserDefaults`, toggled in Settings) switches all
80 books — the 66 canonical books and all 14 Apocrypha/Deuterocanon books (`manifest.json`'s
`hasOriginalLanguage` flag is `true` for every entry) — from the KJV/AV English in `Resources/Bible`
to Hebrew (OT), Koine Greek (NT + most Apocrypha), or Latin (2 Esdras) in `Resources/BibleOriginal`
— same per-book JSON schema (`{"book", "chapters": [{"chapter", "verses": [...]}]}`) either way, so
`BibleBook`/`BibleStore`'s resolution logic doesn't care which language it loaded. Verse/chapter
numbering in `BibleOriginal` is deliberately KJV-aligned (not the source language's native
versification) wherever a Lesson reference could actually cite it by chapter:verse, so
`ScriptureRef`-based lookups keep working unmodified; a couple of Apocrypha books (e.g. Additions to
Esther) instead use the source's own natural verse divisions because nothing in the lectionary cites
them by chapter:verse and forcing KJV-style alignment would have required guessing at ambiguous
verse-boundary splits between disagreeing textual traditions. See `Scripts/convert_original_language_bible.py`'s
module/function docstrings for the source-by-source reasoning (recension choices, versification
crosswalks, known textual-tradition gaps between e.g. SBLGNT and the Textus Receptus, or the
Vulgate's historically-lost-then-rediscovered fragment of 2 Esdras 7).

## Content sourcing conventions

Nearly every piece of liturgical/scriptural content in this app was transcribed or parsed from a
specific, cited, publicly-accessible source (see `README.md`'s "Content notes" and per-feature
sections) rather than generated from training knowledge — when adding or fixing content, find and
cite a source the same way, and prefer writing a small one-off parser (kept in `Scripts/` or a
scratchpad, not shipped) over hand-transcribing large amounts of text.
